# Evaluación Proyecto de Diseño de Software

## Evaluación Proyecto de Diseño de Software (20%)

---

# Contexto General

En esta evaluación, los grupos de trabajo deberán presentar y defender un proyecto de software desarrollado durante el curso.

Se evaluará el avance del proyecto, presentando un Producto Mínimo Viable (MVP), evidenciando cómo el sistema ha evolucionado a lo largo del tiempo en términos de:

- Diseño.
- Implementación.
- Toma de decisiones.

El objetivo principal es demostrar:

- Correcta aplicación de principios de diseño.
- Uso de patrones de diseño.
- Implementación de funcionalidades básicas.
- Construcción de software mantenible y estructurado.

## Fecha de Entrega

La entrega y presentación del proyecto (PPT) se realizará durante la:

- Semana 9.
- Fechas: 11 al 15 de mayo.

---

# Objetivo de la Evaluación

La evaluación busca medir la capacidad del grupo para:

- Mostrar el avance progresivo del proyecto.
- Aplicar principios de diseño de software.
- Aplicar principios SOLID.
- Justificar decisiones de diseño.
- Incorporar patrones de diseño correctamente.
- Construir un sistema coherente.
- Construir software mantenible y estructurado.
- Desarrollar código utilizando APIs.

---

# Requisitos del Proyecto

# 1. Avance del Proyecto

El grupo deberá demostrar claramente la evolución del sistema.

## Aspectos requeridos

- Estado inicial del proyecto.
- Cambios realizados durante el desarrollo.
- Mejoras incorporadas mediante retroalimentación.
- Decisiones de diseño tomadas durante el proceso.

---

# 2. Aplicación de Principios SOLID

El sistema debe evidenciar la correcta aplicación de los principios SOLID.

## Principios requeridos

- SRP — Single Responsibility Principle.
- OCP — Open/Closed Principle.
- LSP — Liskov Substitution Principle.
- ISP — Interface Segregation Principle.
- DIP — Dependency Inversion Principle.

## Requisito

Los principios deben:

- Estar implementados.
- Ser correctamente justificados.

---

# 3. Uso de Patrones de Diseño

El proyecto debe incluir como mínimo:

## Patrones requeridos

- 1 patrón creacional.
- 1 patrón estructural.
- 1 patrón de comportamiento.

---

# 4. Ejecución del Software

El grupo deberá:

- Mostrar el sistema funcionando.
- Ejecutar funcionalidades básicas.
- Ejecutar requerimientos.
- Evidenciar principios SOLID en ejecución.
- Mostrar el uso de patrones de diseño.

---

# 5. Uso de APIs

El sistema debe integrar APIs cuando sea pertinente.

## Tipos de APIs permitidas

- API externa.
- API propia.
- API interna.

## Requisito

Debe justificarse:

- Su uso.
- Su diseño.

---

# 6. Justificación de Diseño

El grupo deberá explicar:

- Arquitectura del sistema.
- Decisiones tomadas.
- Evolución del sistema.
- Mejoras implementadas.

---

# Informe del Proyecto

El informe debe incluir:

- Nombre del proyecto.
- Integrantes.
- Problema abordado.
- Avance del proyecto.
- Requerimientos.
- Reglas de negocio.
- Principios SOLID.
- Diagramas SOLID.
- Patrones de diseño.
- Diagramas de patrones.
- APIs utilizadas.
- Backlog.
- Participación del equipo.

---

# Presentación del Proyecto

La presentación debe realizarse utilizando:

- PowerPoint (PPT).

## Contenido obligatorio

- Descripción del problema.
- Arquitectura.
- Principios SOLID.
- Patrones utilizados.
- Evolución del sistema.
- Demostración en vivo.

---

# Entregables

## Archivos requeridos

1. Código fuente (GitHub).
2. Informe.
3. Backlog (Trello u otro).
4. Presentación PPT.
5. Demostración del sistema.

## Ponderación

- Informe: 60%.
- Presentación: 40%.

# Recomendaciones Técnicas para el Proyecto

## Recomendaciones sugeridas

Para maximizar la calidad del proyecto y alinearse con la rúbrica, se recomienda:

- Utilizar arquitectura limpia (Clean Architecture).
- Mantener separación de responsabilidades.
- Documentar decisiones técnicas.
- Justificar patrones de diseño.
- Mantener un backlog actualizado.
- Integrar diagramas UML.
- Utilizar Git y GitHub adecuadamente.
- Mostrar evolución progresiva del sistema.

## Recomendaciones para la presentación

- Mostrar funcionalidades reales.
- Realizar una demo estable.
- Explicar SOLID con ejemplos del propio código.
- Mostrar patrones aplicados en clases reales.
- Relacionar requerimientos con implementación.
- Preparar respuestas técnicas sobre arquitectura y decisiones.
